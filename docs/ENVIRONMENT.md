# Environment Configuration

This file documents runtime environment variables that affect public-facing
behavior, with a focus on the variables whose absence causes a feature to
silently degrade. The canonical list of all variables (with descriptions) is
`.env.example` at the repo root.

---

## Vercel Production setup — generic walkthrough

Every environment variable below follows the same Vercel UI workflow.
Treat any change to these as a **deploy event**, not a settings tweak: a
Vercel serverless function only sees a new env value after the next build
completes. **No code change is required to add or rotate a key**, but a
redeploy is.

This section is purely **manual / external** — none of these steps are
in the code.

### 1. Open the Vercel project

1. Sign in at <https://vercel.com/dashboard>.
2. Pick the **ConstructAIQ** project.
3. **Settings** (top nav) → **Environment Variables** (left rail).

### 2. Add or update the variable

1. Click **Add New** (or click an existing variable's `…` → **Edit**).
2. Set the **Key** to the exact env name (e.g. `ANTHROPIC_API_KEY`,
   case-sensitive — Linux Vercel runtime is case-sensitive).
3. Paste the **Value**. Vercel masks the field after save; the value
   is never shown in plaintext in the UI again.
4. **Environments** — tick **Production**. For dev parity also tick
   **Preview** and **Development**, but Production is the only one
   that affects `constructaiq.trade`.
5. Click **Save**.

> **Never paste a key into a comment, commit message, Slack thread, or
> support ticket.** If a key is pasted anywhere git-tracked, treat it
> as compromised and rotate it.

### 3. Redeploy so the new value takes effect

Existing serverless functions hold the old env until they're rebuilt.
Trigger a redeploy any of these ways — pick whichever is most
convenient:

- **Vercel UI** — Deployments → newest **Production** deployment → `…`
  menu → **Redeploy**. Leave **Use existing Build Cache** ticked.
- **Git push** — push any commit to `main` (a no-op `docs:` commit is
  acceptable).
- **Vercel CLI** — `vercel --prod` from a clean checkout.

The deploy needs to reach the **Ready** state. A new env value picked
up by a still-building deployment will not be live until that build
finishes.

### 4. Verify Production picked up the variable

`/api/status` exposes a single safe boolean per critical env var (no
key values are ever returned). After the redeploy lands:

```bash
# Required infrastructure — every value must be true before declaring launch-ready
curl -s https://constructaiq.trade/api/status | jq .env
# expected: {
#   "supabaseConfigured":   true,
#   "anthropicConfigured":  true,
#   "upstashConfigured":    true,
#   "sentryConfigured":     true,
#   "cronSecretConfigured": true
# }

# Optional data-source keys — `true` means the route can fetch live; `false` is the documented degraded fallback.
curl -s https://constructaiq.trade/api/status | jq .api_health
```

If the boolean is still `false` after a redeploy:

1. Re-open the Vercel env settings and confirm the variable is
   **scoped to Production** (not just Preview).
2. Confirm the value is **non-empty** (Vercel preserves whitespace
   exactly — a stray newline or space changes the value).
3. Inspect the Function logs for the latest deployment for any
   `[scope]` log line written by the observability helper
   (`src/lib/observability.ts`). Stable scopes that surface env
   issues: `[federal]`, `[weeklyBrief]`, `[status]`,
   `[pricewatch]`, `[forecast]`, `[cshi]`, `[dashboard]`.

### 5. Local development equivalent

For development parity, mirror Production values into `.env.local`:

```bash
cp .env.example .env.local
# Edit .env.local — paste the same values you set in Vercel Production
npm run dev
curl -s http://localhost:3000/api/status | jq .env
```

`.env.local` is git-ignored. Never commit it.

## ANTHROPIC_API_KEY — required for live Weekly Brief generation

The Weekly Brief is generated by Claude (`@anthropic-ai/sdk`) at the
`/api/weekly-brief` endpoint and is also produced on a weekly cron at
`/api/cron/brief`. Both code paths require `ANTHROPIC_API_KEY`.

### Behavior when unset

| Surface              | Behavior                                                                 |
|----------------------|--------------------------------------------------------------------------|
| `/api/weekly-brief`  | Returns `source: "static-fallback"`, `live: false`, `configured: false`. |
| Brief body           | Neutral message — no numeric claims, no stale CSHI/permit/award figures. |
| `/api/cron/brief`    | Returns `status: "fallback"`, no row inserted into `weekly_briefs`.      |
| `/api/status`        | Reports `weekly_brief.configured: false`.                                |
| `/api/dashboard`     | `brief_excerpt` is `null` whenever no real AI brief is cached.           |
| Dashboard UI         | `WeeklyBrief` component renders an `UNAVAILABLE` badge for the fallback. |

The fallback intentionally does **not** present fabricated content as live —
the previous static brief contained specific numeric claims that were stale
the moment they were committed and is no longer used.

### Setting the key in Vercel

1. Generate a key at <https://console.anthropic.com/settings/keys>.
2. Vercel project → **Settings → Environment Variables**.
3. Add `ANTHROPIC_API_KEY` with the new value, scope **Production** (and
   optionally **Preview** + **Development**).
4. Trigger a redeploy (`vercel --prod` or push to `main`) so the new env is
   picked up by serverless functions.

### Verification

```bash
# Should report true
curl -s https://constructaiq.trade/api/status | jq .env.anthropicConfigured

# Should return source: "ai", live: true (after the first generation)
curl -s https://constructaiq.trade/api/weekly-brief | jq '{source, live, configured}'
```

If the brief still returns `source: "static-fallback"` after a redeploy, check
that `ANTHROPIC_API_KEY` is non-empty in the Production scope and that the
function logs do not show a Claude API error (rate limit, invalid key, etc.).

### Local development

```bash
cp .env.example .env.local
# Edit .env.local and paste your ANTHROPIC_API_KEY value
npm run dev
curl -s http://localhost:3000/api/status | jq .env.anthropicConfigured
```

The key value is never logged or returned by any API surface. Only the
`configured` boolean is exposed via `/api/status`.
